# 项目演示视频讲稿

建议视频时长控制在 7 到 9 分钟。录制时依次展示 README、Tableau、最终报告图片和代码目录，不需要逐行讲代码。

## 0:00–0:40 项目介绍

大家好，这是我的 Amazon 电商商业数据分析项目。项目周期为四周，目标是完成从数据理解、清洗、探索性分析，到文本建模、推荐原型和商业建议的完整流程。

项目主要回答产品类目、用户行为、评论关注点和推荐策略四类问题。同时我也特别区分了数据能够直接回答的问题、只能使用代理指标的问题，以及当前无法回答的问题。

## 0:40–1:30 数据与限制

项目使用两份数据。第一份包含 1,465 条产品记录，用于类目、价格、评分和评论分析；第二份包含约 782 万条评分行为，用于用户趋势和推荐模型。

两份数据只有 6 个商品 ID 重合，因为它们来自不同时间和商品池，所以我没有强行合并。项目也没有真实订单和库存数据，因此商业关注权重不是真实销售额，高活跃用户也不能直接称为高价值用户。

## 1:30–2:20 数据处理流程

第一周我检查了缺失、重复、异常和字段类型，并完成价格、折扣率、评分数、时间和类目字段的清洗。这里可以快速展示 `src/data/`、数据字典和 Week 1 输出。

第二周我生成了 Tableau 可以直接使用的 CSV，并完成类目、产品、用户和年度趋势分析。

## 2:20–3:20 商业概览

打开 Business Overview 仪表盘。说明 Electronics 占估算商业关注权重的 82.3%，是当前样本中最需要关注的类目。但这个指标由折扣价乘评分数量得到，只表示关注优先级，不是真实 GMV。

再展示重点优化产品图：100 个机会产品中 Electronics 有 51 个，Home&Kitchen 有 25 个，Computers&Accessories 有 23 个。因此资源优先级应从 Electronics 开始。

## 3:20–4:10 用户行为

展示用户活跃度分布。约 68.6% 的用户只评分一次，说明评分矩阵非常稀疏。

展示年度趋势。评分活动在 2013 年达到峰值，但 2014 年可能是非完整年份，所以不能直接说业务下降。

高活跃用户的评分整体更正向，但由于没有订单金额，我把他们定义为高互动人群，建议用于新品试用和评论邀请，而不是直接当作高消费用户。

## 4:10–5:35 NLTK 与 LDA

首先展示词典 baseline 与 NLTK VADER 对比。两种方法一致率为 91.26%，VADER 识别出 74 条负向记录，比 baseline 的 26 条更多，说明 VADER 对负面表达更敏感。

接着展示 LDA。模型比较了 3 到 8 个主题，最终选择验证集困惑度最低的 3 主题模型。

三个主题分别是易用性与性价比、智能设备与音频、充电线缆与连接。其中易用性与日常体验覆盖 56.72% 的记录，负向比例也是最高的；智能设备与音频平均评分最低，因此产品改进应重点关注安装、续航、屏幕、声音和蓝牙连接体验。

## 5:35–6:25 推荐系统

展示推荐评估图。item-item 共现模型生成了 4,862 条推荐关系，在 5,000 名用户上的 Hit Rate@5 是 7.12%，Hit Rate@10 是 8.64%。

这个结果说明模型可以作为候选商品召回 baseline，但评分矩阵稀疏，效果仍然有限。后续应加入商品属性、热门商品兜底、协同过滤和线上 A/B 测试。

## 6:25–7:40 商业建议

第一，建立产品风险清单，优先排查 Electronics 的重点机会产品，并结合评分、VADER 情感和 LDA 主题判断风险原因。

第二，优化商品页面。易用性主题应加强安装和使用说明；智能设备应说明续航、屏幕和连接稳定性；充电类商品应统一展示接口、功率和兼容设备。

第三，把高活跃用户用于新品试用和反馈运营。获得订单数据后，再建立真正的 RFM 或 CLV 高价值用户分群。

第四，当前不能直接做库存预测。下一步需要接入 SKU 级销量、库存、促销、退货和交付周期数据，再建立时间序列模型。

## 7:40–8:20 项目结构与结尾

最后展示 GitHub README 和项目结构。`src/` 保存可复现代码，`outputs/` 保存分析表，`reports/` 保存 Tableau 和图片，`docs/` 保存周报和最终报告。

这个项目的核心价值是建立了完整分析闭环，同时没有为了回答业务问题而超出数据能够支持的范围。谢谢观看。
